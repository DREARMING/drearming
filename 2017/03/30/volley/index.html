Volley网络框架

慢慢体会一个网络框架的构造过程

Posted by MVCoder on March 30, 2017

Volley框架

Volley 是 google 2013年io大会推出的网络请求库，非常适合网络请求频繁，数据量不大的需要。对于大文件请求，则性能不太好

###Volley的用法

Volley先调用它的静态方法 newRequestQueu(context)，初始化我们整个Volley最重要的对象 RequestQueue，我们看看newRequestQueue 做了一些什么；

我们发现这个方法有几个重载方法，最后都回到到上面这个方法。 在这个方法里面，我们新建了一个Volley的缓存目录，构造了网络请求对象， 然后调用了RequestQueue的构造方法，传进DiskBasedCache和network 对象，构造出一个 queue对象，调用其start方法后，返回。。

对于Network，是一个简单的接口，要求我们的实现类，必须实现这个方法

对于 basicNetWork,我们就暂时不深入了解，我们就知道，它执行传进来的Request，返回一个NetworkResponse 对象；

再来看看 DiskBasedCache(cacheDir,maxDiskCacheBytes) ,这个类实现了一个Cache接口

这个接口定义了 我们一个Cache对象，需要实现的一些方法，get、put、remove、clear..

我们看一下这个接口里面的内部类 Entry，它定义了作为一个Cache应该具有的特性：

  1. 缓存会过期 ttl
  2. 缓存会丢失
  3. 缓存需要刷新 soft_ttl
  4. 缓存具有数据 data:byte[]
  5. 缓存是否修改过 last_modified

我们看一下DishBasedCache这个磁盘缓存类的一些重要方法；

get方法：根据cacheKey来获取缓存的内容，通过cacheKey可以映射到cache目录一个的缓存文件；

我们发现一个CacheHeader 这么一个类，而不是我们上面Cache接口里面的 Entry，其实这里面做了一些优化。。 CacheHeader跟Cache.Entry 没有多大差别，就是缺了一个data数据，而多了一个long类型的size，代表data：byte[]的大小； 因为我们可以想象一下，这样做的好处是：Cacheheader拥有了Cache的特性，我们可以用它来做缓存所有的东西，仅当我们需要获取缓存的数据时才将数据加载到内存中，这样子可以有效避免OOM；当然缺点也是有，读取缓存的速度慢了，但是我觉得这是非常值得。

其他的过程就很简单了，就是将CacheHeader转化为Cache.entry返回，这里涉及了一些Volley设计缓存读写的一些简单规则，有兴趣的可以看一下。

put方法

很简单的一个方法，首先检查当前的缓存容量是否足够添加这个entry，调用pruneIfNeeded方法

如果可以容纳下这个cache数据的话，就直接返回，否则的话，就遍历当前的 cacheHeader集合，开始删除一些缓存文件，直到缓存容量可以容纳刚刚传进来的cache大小为止；

回到put方法，把cache.entry对象的数据存到一个缓存文件中，然后构造它的替身cacheHeader，存放到 mEntries 这个集合中；

好了，DiskBasedCache 就讲这两个方法，其他的有兴趣可以自己看一下Volley的源码；

RequestQueue

我们看看它的几个非常重要的成员变量：

Cache 我们可以从构造方法，可以很容易看出，就是在Volley的静态方法 newRequestQueue 被调用的时候，传递过来的DiskBaseCache对象，上面我们已经分析过了，是一个磁盘的缓存对象；

mNetWork 同样也是 newRequestQueue传递过来，用于将一个Request对象 执行网络请求，并将返回结果封装成 NetworkResponse 对象返回。

mDispatchers = new NetworkDispatcher[threadPoolSize] 构造方法有这么一条语句，构造了一个NetworkDispatcher的数组，我们看看NetworkDisPatcher究竟做了什么？

我们发现，NetworkDispatcher 继承与 Thread，RequestQueue构造的时候将 Cache，mNetwork，delivery，还有 一个BlockingQueue,用于存放网络任务；

从Run方法中我们就知道，NetworkDispatcher的作用就是 不断从 一个BlockingQueue中读取 Request， 读取到一个Request将执行以下一些策略：

  1. 先判断 Request 是否已经取消，如果已经取消，就直接返回，不用进行网络请求
  2. 对 Request 执行网络请求，并返回封装好结果的 NetworkResponse 对象
  3. 如果 NetworkResponse 对象显示 结果并没有发生过改变或者当前请求已经发送结果到客户端了，直接返回，不需要进行其他操作；
  4. Volley的请求 跟 返回结果密切相关，请求中定义了 结果转化的方法 parseNetworkResponse，可以将 结果对象转化为我们 请求需要的 结果类型
  5. 然后判断当前的请求是否需要缓存，如果缓存的话，就用cache 来做缓存
  6. 将 当前的 request 和 response 发送回主线程，通过 mDelivery.postResponse方法

我们在RequestQueue 的构造方法里面看到这么一条语句，传进一个Handler对象作为构造参数，并且这个Handler关联了主线程

再看看ExecutorDellivery的内部实现

很容易就看出，当我们调用postResponse方法时，都是通过 handler将 Runnable 返回到主线程后执行；

postResponse方法里面，将 Request 和 Response 封装在 ResponseDeliveryRunnable 的Runnable对象中，当我们将这个Runnable发送回到主线程，做了下面这些策略：

  1. 判断当前请求是否取消了？取消就不回调接口了
  2. 判断当前 response 是否是成功的，如果成功的话，就调用 请求的 deliverResponse（result）方法， 这是一个Request的一个抽象方法，我们拿 StringRequest 做一个例子：

我们构造 StringRequest方法时，就传进去了一个Response.Listener，这个时候，经历种种，最后回调到这里。如果发生错误，就会回调它的 onErrorResponse 方法；

CacheDispatcher

我们回到 RequestQueue 中，它还有一个 CacheDispatcher 成员变量，它有什么作用呢？从上面的NetworkDispatch可以猜出，这个家伙肯定也是一个线程，用来读取 Cache 的BlockingQueue，然后对请求作一些处理和分发。 没错，就是这样子的

同样，我们来分析下 CacheDispatcher 的 Run方法，当它从 缓存请求的 BlockingQueue 里面读到一个 Request 时，做的一些策略：

  1. 首先判断请求是否取消了，取消就不用继续了，直接读取下一个请求。
  2. 根据 请求，查找 cache中是否有 这请求的 缓存结果，如果没有的话，就将这个请求塞给给 网络请求的 BlockingQueue，让NetWorkDispatcher处理；
  3. 查找到缓存结果，判断这个缓存是否已经过期？如果过期，将请求塞给 网络请求的 BlockingQueue
  4. 我们将 缓存的 Cache.Entry 转化成 NetworkResponse 对象，然后通过 Request请求对象将其转化成 我们请求的结果类型，返回一个Response 对象
  5. 如果当前的 response 不需要刷新的话，直接通过 mDelivery 发送会主线程，这个过程在上面已经说过
  6. 如果需要刷新，即 超过 soft ttl时间，则同样通过mDelivery发送 request，response回主线程，但是这里传多了一个Runnbale，就是结果返回后，将这个请求添加当 网络请求的BlockingQueue，用于更新现在的缓存；

RequestQueue 的重要方法

到上面，我们已经几乎了解RequestQueue 的一些重要成员变量，我们来看看 它的start方法

很简单，就是将我们的 NetWorkDispatcher 和 CacheDispatcher线程运行起来，这个时候 这些用来处理请求的线程对象，分别不断读取着 mNetworkQueue（网络请求 BlockingQueue），mCacheQueue（缓存请求 BlockingQueue），当这两个BlockingQueue一旦有请求添加进去，就会被分发处理

我们来看一下如何添加请求？

添加一个请求做了下面这些策略：

  1. 设置当前RequestQueue在这个请求里面， 当request调用finish方法后，就会标记当前请求已经完成处理（包括请求取消后返回，获取返回结果返回等等情况），然后回调 RequestQueue 的finish（Request）方法，标记当前运行中的所有请求，这个请求已经完成，应该移除；
  2. 将添加进来的请求 塞进 当前RequestQueue 的 mCurrentRequests，表示这个请求还没有处理；
  3. request.setSequence(getSequenceNumber()) 为请求添加一个Sequence，这是因为BlockingQueue是PriorityBlockingQueue，里面有个CompareTo方法需要实现，用来对 Request 进行排序；
  4. 如果当前请求不需要 缓存， 则直接添加到 网络的BlockingQueue中，让NetworkDispatcher来分发处理；
  5. 对需要缓存的请求进行一些策略处理。。
    1. 因为我们可能手抖或者设备比较渣，或者操作太快了，连续发送了两次相同的请求，这个时候，我们可以认为，为了高效的请求处理，我们只需要请求一次，然后接下来相同的请求读取缓存就好，这样就加快我们的处理速度了。。所以我们会想到，将具有相同 cacheKey 的请求 弄成一个链表，不急着将他们一下子分发出去，只要等待和当前请求相同的cacheKey的请求处理 finish 后再发送也不迟；
    2. 如果没有相同的cacheKey请求，则发送这个请求到 cache的BlockingQueue中，让 cacheDispatcher进行分发

上面说过finish方法，看一下 request完成后回调 RequestQueue 的 finish方法，做了一些什么？

可以很容易就看出了，就是将当前请求移除 RequestQueue正在处理的列表，并且调用 监听接口；同时，上面说过的 mWaittingRequests，那个链表式 缓存请求分发机制，这里当第一个Request完成后，后面具有相同的 cacheKey的所有请求，马上塞到cache的BlockingQueue中，进行分发处理。

当然，我们请求的必须具有取消的功能啊？上面 netWorkDispatcher 和 CacheDispatcher 都做了request是否取消的判断，那么来看看RequestQueue的cancel方法

比较简单，遍历 RequestQueue 的 mCurrentRequests，符合传进来实现了RequestFilter这个接口的对象的apply方法规则的requst进行取消；

比起语法更加重要的是，要理解应用场景，我们一般什么时候会取消请求呢？然后怎么使用Volley来进行取消呢？

举个栗子：用户的操作是拥有无限可能的，比如我们的Activity进行打开过程中，加载了一些文字，网络的图片准备加载了，用户发现尼玛，我怎么用着流量看这些高清美女大图，赶紧关了activity，开个wifi再来看或者觉得不应该看了，是时候面对现实出手了。。

这个时候我们有理由认为，当用户关掉activity的时候，用户不想看那些图片，绝不应继续进行网络访问了，你如果有种，关了还在不停刷用户的流量，你简直在作死。。 这个时候应该调用 RequestQueue的cancelAll方法，我们传递的tag，我们可以将当前Activity的名字作为tag传进来，每一个请求被创建的时候都可以指定一个tag，当然还有各种各样的方式，自己去diy；然后再activity的onStop方法或者onDestroy方法，调用cancelAll；

好了，到此，Volley整个框架的调用过程就清晰了；

我们最后总结一下整个过程：

  1. 一开始调用Volley.newRequest(context)方法，创建了一个 Cache的缓存目录，构造了一个DiskBasedCache对象，用于进行请求结果的缓存（磁盘）； 然后根据当前api版本，创建了一个 NetWork对象，用来处理请求和封装结果； 我们还构建了RequestQueue方法，调用了其start方法
  2. RequestQueue 有几个非常重要的成员
    1. mNetwork 用于网络访问，这里主要是将它作为构造参数 传给 NetworkDispatcher，进行网络请求和封装结果
    2. mCache 上面构造的DiskBasedCache对象，传给 CacheDispatcher 和 NetworkDispatcher作为构造参数，用来处理 Request的缓存操作
    3. NetWorkdispatcher 数组，是一个 Thread对象，轮循 mNetworkQueue，处理从里面读出来Request，处理的策略上面有介绍过：
      1. 先判断 Request 是否已经取消，如果已经取消，就直接返回，不用进行网络请求
      2. 对 Request 执行网络请求，并返回封装好结果的 NetworkResponse 对象
      3. 如果 NetworkResponse 对象显示 结果并没有发生过改变或者当前请求已经发送结果到客户端了，直接返回，不需要进行其他操作；
      4. Volley的请求 跟 返回结果密切相关，请求中定义了 结果转化的方法 parseNetworkResponse，可以将 结果对象转化为我们 请求需要的 结果类型
      5. 然后判断当前的请求是否需要缓存，如果缓存的话，就用cache 来做缓存
      6. 将 当前的 request 和 response 发送回主线程，通过 mDelivery.postResponse方法
    4. CacheDispatcher 也是一个Thread对象，轮循 mCacheQueue，处理从里面都出来的 cache request，处理的策略上面同样介绍过：
      1. 首先判断请求是否取消了，取消就不用继续了，直接读取下一个请求。
      2. 根据 请求，查找 cache中是否有 这请求的 缓存结果，如果没有的话，就将这个请求塞给给 网络请求的 BlockingQueue，让NetWorkDispatcher处理；
      3. 查找到缓存结果，判断这个缓存是否已经过期？如果过期，将请求塞给 网络请求的 BlockingQueue
      4. 我们将 缓存的 Cache.Entry 转化成 NetworkResponse 对象，然后通过 Request请求对象将其转化成 我们请求的结果类型，返回一个Response 对象
      5. 如果当前的 response 不需要刷新的话，直接通过 mDelivery 发送会主线程，这个过程在上面已经说过
      6. 如果需要刷新，即 超过 soft ttl时间，则同样通过mDelivery发送 request，response回主线程，但是这里传多了一个Runnbale，就是结果返回后，将这个请求添加当 网络请求的BlockingQueue，用于更新现在的缓存；
    5. mCurrentRequests 存储着当前RequeueQueue中还没处理或者正在处理的所有Request，可以用来进行cancel操作， request回调RequeueQueue的finish方法时，会将这个request从mCurrentRequests中移除；
    6. mWaittingRequest 上面解释过它有什么用了，主要是为了优化具有相同 cacheKey 的一系列请求，等到第一个完成请求后，再把它们添加到mCacheQueue, 那么第一个请求完成后，相同cacheKey的请求就有缓存了，不用频繁进行网络操作，
  3. 调用start方法后，上面的CacheDispatcher和NetworkDispatcher，就会不断读取 BlockingQueue中的请求，整个Volley框架就运行起来了；
  4. 当有 RequeueQueue调用add方法，添加一个请求进来的时候，就会做一系列策略：
    1. 设置当前RequestQueue在这个请求里面， 当request调用finish方法后，就会标记当前请求已经完成处理（包括请求取消后返回，获取返回结果返回等等情况），然后回调 RequestQueue 的finish（Request）方法，标记当前运行中的所有请求，这个请求已经完成，应该移除；
    2. 将添加进来的请求 塞进 当前RequestQueue 的 mCurrentRequests，表示这个请求还没有处理；
    3. request.setSequence(getSequenceNumber()) 为请求添加一个Sequence，这是因为BlockingQueue是PriorityBlockingQueue，里面有个CompareTo方法需要实现，用来对 Request 进行排序；
    4. 如果当前请求不需要缓存，则直接添加到网络的BlockingQueue中，让NetworkDispatcher来分发处理；
    5. 对需要缓存的请求进行一些策略处理。。
      1. 因为我们可能手抖或者设备比较渣，或者操作太快了，连续发送了两次相同的请求，这个时候，我们可以认为，为了高效的请求处理，我们只需要请求一次，然后接下来相同的请求读取缓存就好，这样就加快我们的处理速度了。。所以我们会想到，将具有相同 cacheKey 的请求 弄成一个链表，不急着将他们一下子分发出去，只要等待和当前请求相同的cacheKey的请求处理 finish 后再发送也不迟；
      2. 如果没有相同的cacheKey请求，则发送这个请求到 cache的BlockingQueue中，让 cacheDispatcher进行分发
  5. 最后就是 CacheDispatcher 或者 NetworkDispatcher进行结果分发了，这个时候因为分发措施是消息队列机制，并且是主线程的Handler，所以会调度回主线程，最后会调用我们请求创建过程时，传进来的Response.listener的接口实现对象，成功会调用 它的onResponse（T response），出错则会调用 onErrorResponse(VolleyError)方法；

一些思考：

一个网络请求框架应该具备什么？

  1. 应该具备执行网络请求并且能够返回请求数据的能力。。 即访问网络的模块
  2. 请求的封装能力，可以通过url，请求头这些基本请求信息，来封装成一个请求，这个类应该包含请求的基本性质，即Request类
  3. 不同的请求有不同返回结果的类型，应该具备将请求结果封装成Response，并且它的实际数据内容，应该跟对应请求的返回类型一致，Volley用到的措施是通过Request的抽象方法， abstract protected Response parseNetworkResponse(NetworkResponse response);所有的子类必须实现这个方法，正确的将response转化为自己这种类型请求需要的结果；将返回结果转化成 request 需要的返回类型，这就解决了 请求类型和返回结果类型的一致性问题；
  4. 如果增加请求的速度？多线程分发处理请求，PriorityBlockingQueue的使用
  5. 如何将请求结果返回到主线程？ 因为经常涉及到一些UI工作，必须要在主线程操作，通过Handler分发，此Handler与主线程的Looper相关联
  6. 为了增加访问效率，节省流量，缓存的措施是不可少的。 即缓存Model，包括控制缓存大小和管理读，取，移除；
  7. 请求可以被取消的能力，相同请求的流水线模式，即Volley的mWaittingRequest的优化手段

请求的基本性质：

  1. 请求方法
  2. 请求url
  3. 请求头
  4. 请求参数、正文
  5. 请求的状态（是否被取消，finish）
  6. 需不需要缓存
  7. 优先级
  8. 转换Response为请求结果类型的能力

返回结果的基本：

  1. 状态码
  2. 返回的内容
  3. 返回结果是否有修改，304
  4. 响应头

关于缓存，缓存具备哪些特性

  1. 缓存会过期 ttl
  2. 缓存会丢失
  3. 缓存需要刷新 soft_ttl
  4. 缓存具有数据 data:byte[]
  5. 缓存是否修改过 last_modified

一个缓存Model需要实现的方法

  1. get（cacheKey）
  2. put(cacheKey,object);
  3. remove(cacheKey);
  4. clear();
  5. initialize(); 假如是文件缓存的话，我们可以初始化这个cache model，读取在缓存目录中，当前已经缓存了哪些文件，然后初始化内存缓存；假如只有内存缓存，则不必须了。。 我们可以考虑到，如果将所有缓存文件立刻加载到内存中，这种做法很容易导致oom的，所以可以考虑，存储Cache的Entry类，一个代表类，它包括cache的特性，但是没有正文，正文需要在访问的时候才从文件中读取。
