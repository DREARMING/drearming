我们是如何将一个View显示在window上呢

Posted by MVCoder on March 18, 2017

理解 Window 和 WindowManager

参考：任玉刚 《Android开发艺术探索》 第八章理解Window 和 WindowManager

我们写好的布局文件，通过LayoutInflater,生成一个View对象，添加到ViewRootImpl的DecorView的ContentLayout中，DecorView是视图的顶层类，那么它是如何呈现在用户面前的呢？

首先我们需要有这么一个概念，View的显示并不是当前应用程序操作的，而是系统服务WMS来具体呈现View的，然后我们针对当前窗口的所有操作View的事件，WMS都会跨进程把事件传送回来，供我们使用；WMS是操作所有窗口的系统服务，Window的具体实现也是在WMS中。而在我们客户端的Window对象，其具体实现是PhoneWindow，我们可以简单地理解为是一个具有传递View到WMS和与WMS进行交互的功能类，它具有一个DecorView（就是窗口要显示的View），还有一些回调接口，用于View事件的回调；

我们常常用到的Activity，Dialog还是Toast，它们的视图View实际上都是附加到 Window上，然后WindowManager将View 添加到WMS中；我们来看看WindowManager是如何将View添加到Wms的

WindowManager

Window 有两个参数是比较重要的，分别是 WindowManager.LayoutParams 的 flag 和 type

flag 参数表示Window的属性，控制Window的显示特性，它有很多属性，可以参考官方文档，下面列几个常用而且重要的

FLAG_NOT_FOCUSABLE

表示Window不需要获取焦点，也不需要接受各种输入事件，此标记会同时启用FLAG_NOT_TOUCH_MODAL,最终事件会直接传递给下层的具有焦点的Window;

FLAG_NOT_TOUCH_MODAL

在此模式下,系统会将当前Window区域以外的单击事件传递给底层的Window，当前Window区域以内的单击事件则自己处理。这个标记很重要，一般来说都需要开启此标记，否则其他Window将无法收到单击事件。

FLAG_SHOW_WHEN_LOCKED

开启此模式可以让Window显示在锁屏的界面上

Type参数表示Window的类型，Window有三种类型，分别是应用的Window、子Window和系统Window。应用的Window对应着一个Activity。子Window不能单独存在，他需要依赖在特定的父Window之中，比如常见的Dialog就是一个子Window。 系统Window需要声明权限才能创建，比如系统的状态栏和Toast，如果不声明权限则会报错。Toast不需要声明权限

Window是分层的，应用型在底层（0-999），子Window在中层（1000-1999），系统window在最上层（2000-2999）

WindowManager的方法

WindowManager extends ViewManager，我们来看一下ViewManager的接口方法

上面的三个方法就是我们通过与 WMS 跨进程通信时候，用来操控Window的方法，可以看出来，我们其实都是操作View对象的，而客户端的Window相对来说是一个抽象的概念，它具有DecorView和一些回调接口，具体实现是在Window

addView方法是将一个View对象通过Binder机制，添加到WMS去；这里不具体讨论WMS的过程。

其实更加具体的话，WindowManager只是一个接口，它的具体实现类是WindowManagerImpl

WindowManagerImpl很明显只是一个代理类，具体执行上面几个方法的是WindowMangagerGlobal

WindowManagerGlobal 有四个比较重要的成员变量，缓存着Window的重要数据，可以通过索引获取具体的Value

我们可以看到，当我们addView的时候，将会创建 ViewRootImpl 对象，把涉及到Window的重要数据缓存到上面四个成员变量中，但是这个时候还没有进行显示呢。

到了这里，我就想探讨一下从调用setContentView开始，到底经历过一些什么，才完成View显示到Window这一个过程

首先，我们创建一个Activity时，AMS会发送Launcher_Activity的请求消息到客户端，客户端就会执行 performLaunchActivity 方法,在Activity方法中，通过类加载，将Activity实例化，并且通过attach方法，绑定到上下其需要的上下文

其中传进一个window对象，在attach方法里面，会帮助我们实例化window对象，并且设置window的回调接口对象为当前activity，activity实现了Window的Callback接口。这样子，当WMS传回事件时，Window就可以将其传回到activity去；

当我们调用setContentView方法时，其实调用的是PhoneWindow的setContentView方法：

  1. 如果decorView == null 的话，就调用 installDecor()方法的内部的 generateDecor() 方法来构建Decorview,为了初始化DecorView，PhoneWindow还需要通过generateLayout方法来加载具体的布局文件到DecorView中，具体的布局文件和系统的版本以及主题有关

  1. 将 LayoutInflater inflate出来的 View（即我们应用的布局）添加到DecorView的mContentParent中，这就是为什么方法名叫setContentView，而不是View了，我们真正的顶级View是DecorView；

  2. 回调activity的onContentChange 方法，通知activity的视图已经发生改变

  3. 到此为止，PhoneWindow的decorView已经实例化完成了，但是并不能够被WindowManager识别，因为还没有调用WindowManager add这个decorView，那么什么时候会WindowManager会将其呈现出来呢？ 很容易联想到activity的onResume方法，最后会调用activity的makeVisible()方法

这里，这个过程就清晰了：

  1. 我们activity初始化的时候，就已经添加了一个PhoneWindow对象
  2. 通过setContentView方法，实际是调用PhoneWindow的setContentView方法，这里如果是第一次初始化的话，会创建DecorView，加载decorView的布局，然后将我们的布局添加到decorView的mContentParent中，这个时候PhoneWindow的decorView 就创建完毕。
  3. 我们调用onResume方法，真正把DecorView 通过Binder机制 发送给WMS，WMS将其呈现在我们手机屏幕上

不太懂activity的创建原理，可以看我的另外一篇博客：app的启动流程

Window 的删除过程

我们上面已经知道WindowManager调用 三大操作: addView(View,Parameters) removeView(View) updateLayoutView(View,Parameters) 最后都是调用WindowManagerGlobal的同名方法

WindowManagerGlobal 的 removeView方法：

先在WindowManager中找出当前View的下标，然后调用removeViewLock方法

在RemoveViewLocak方法里面，会调用ViewRootImpl类的doDie方法来调用移除这个View

看最后一行，就是调用WindowManagerGlobal的doRemoveView方法，跨进程让WMS移除此View；

window的更新过程

上面在屏幕中添加了一个TextView，在移动的过程中，我们不断更新它的位置，通过updateViewLayout(view,params)方法

我们来看一下

很简单的一个过程，先判断一下参数的合法性，然后更新一下缓存的LayoutParameters，然后调用ViewRootImpl的setLayoutParams（LayoutParams,boolean)方法，这个方法会调用scheduleTraversals() 方法， 这个方法应该很熟悉了，对View重新布局，包括测量、布局、重绘这三个过程。除了View本身的重绘之外， ViewRootImpl 在 performTraversal 方法中会调用relayoutWindow方法，这个方法会通过 mWindowSession，把视图更新到WMS去；mWindowSession的类型是IWindowSession，是一个Binder对象，它是WindowManagerGlobal真正用来实现跨进交流的Binder对象

Dialog 的Window创建

原理基本一样，在Dialog的构造方法里面创建window对象和将dialog作为回调接口

通过show方法，将dialog的decorview添加到window中，然后跨进程显示到WMS去

需要注意的是： dialog的context必须是activity的context，如果是application的context就会报错，dialog是属于子窗口，必须要附着在父窗口上。

Toast

Toast是属于系统级窗口，因为它具备定时取消的功能，所以它略微有点复杂，定时取消的功能是通过Handler的定时机制操作的； Toast涉及两种IPC交流，当我们 调用方法

Toast.makeText(context,"",duration).show();

makeText方法中，加载Android内部一个layout，叫transient_notification的布局，然后将要显示的内容作为布局中TextView的text；总的来说就是构建了一个View对象

看一下show()方法

我们看到 通过INotificationManager将 Toast的 TN 入队；INotificationManager 又是一个Binder对象，它对应的服务是 NWS（NotificationManagerService）,NWS负责统一管理Notification，toast是其中一种，它拥有一个队列根据duration来排队，当出队的时候会调用TN接口的show方法

这里检查了当前应用发送的toast有没有超过50个，如果超过就不入队了，否则就创建ToastRecord，这是为了防止Dos攻击

我们从队列中取出一个ToastRecord就会调用此方法，在这里我们可以看见它做了两件事情：

  1. 它回调了callback的show方法，callback就是我们在客户端的Toast.show()方法时传过来的 TN 对象；
  2. 然后调用scheduleTimeoutLocked(record)；方法，通过Handler机制，过了duration时间段后，这个方法会回调TN对象的hide方法；

我们返回来看看TN是什么？

我们可以从继承关系中很容易发现，TN是一个Binder对象，它内部有一个Handler对象（所以toast必须是在调用了Looper.prepare（）的线程中运行），这也就是为什么NMS能够回调TN的接口，因为TN就是一个Binder对象

当我们从Binder线程池中读到 来自 NMS的回调时，我们这个时候是在Binder线程池的线程中，所以要通过hander来回调回原来的线程

到这里又非常明显了，通过WindowManager来将 Toast的 View 发送到 WMS 去显示。。这里是没有window回调的。我们还记得NMS作了两个操作，一个是回调TN的show方法，过了duration后，又会回调TN的hide方法，这个时候将用windowManager移除掉view；

到此完成了toast的显示和duration时间后取消toast的显示
