Retrofit原理

带你揭开retrofit的神秘面纱

Posted by MVCoder on February 10, 2017

Retrofit基本用法：

动态代理的源码：

//动态代理模式下，对接口方法的调用会转向对InvocationHandler的调用，retrofit很巧妙地利用了这点，在接口的方法(即执行http请求)被调用时，利用传进来的method对象的annotion，reponseType，parameterType构造出一个ServiceMethod对象，这个ServiceMethod对象含有接口方法上的信息（请求头，请求url，请求正文等信息），serviceMethod对象执行toRequest(Object… args)方法后生成 okhttp的 Request对象，实现了接口方法到 okhttp Request的映射，同时 serviceMethod 因为知道返回类型，可以调用toResponse（ResponseBody）将okhttp的responseBody转换为我们的返回类型；

ServiceMethod，它还拥有一个CallAdapter,这个CallAdapter从源代码中可以知道是根据 serviceMethod 的returnType来决定用哪种CallAdapter来回调请求。在默认情况下，retrofit生成一个CallAdapter用于将请求结果回调在主线程中执行。

retrofit 在构造的时候，可以通过

来添加 CallAdapterFactory，这个工厂根据返回类型，决定是否能够处理当前返回类型的请求回调。比如默认的callAdapterFactory则是处理 Call 类型的返回结果，而上述代码是添加了 RxJava2的callAdapter，当接口方法的返回类型是 Observable、Flowable等 Rxjava2 支持的返回类型时，则会返回 Rxjava2CallAdapter 来处理回调。 所以我们应该在工厂方法里面处理自己能够处理的返回类型，不能处理的返回类型会交由下面的 CallAdapterFactory处理。

在默认的情况下，这个 callAdapter 的 adapt 方法是返回一个ExecutorCallBackCall<>对象，并且这个方法的参数，就是一个OkHttpCall<>对象，它里面内部是通过代理的方式，将对okhttpCall的enqueue等操作，做一次代理，会切换到主线程去执行回调。

当然RxJava的callAdapter则会对 okHttpCall enqueue等方法的调用，定义不一样的回调，有兴趣的自己查看。

到了这一步，我们就执行到我们常用步骤的：

上面的call对象就是ExecutorCallBackCall对象，它有一个成员变量就是oktthpCall对象（ServiceMethod方法和arg传进区构造的），当调用它的enqueue方法，就是调用okhttpCall对象的enqueue方法，这个方法里面，可以获取由我们接口设置的真是http请求，下面是真正将serviceMethod转换为HTTP 请求的方法

就可以构造Call对象了，这是 okhttp的 Call了，接下来通过它来调用 okhttp 请求网络，这就成功的将 接口方法切换成 一次网络请求了。

通过请求服务器，返回了 okhttp类型的 Response对象，我们上面提及过，serviceMethod里面可以将ResponseBody转换为我们想要的 returnType类型，因为我们在实际开发过程中，服务器的返回结果，我们往往只想要成功的时候返回我们需要的结果即可，而不用去判断响应码是否为200，这些应该自动化处理掉。

ResponseBody bufferedBody = Utils.buffer(rawBody); return Response.error(bufferedBody, rawResponse); } finally { rawBody.close(); } } if (code == 204 || code == 205) { rawBody.close(); return Response.success(null, rawResponse); } ExceptionCatchingRequestBody catchingBody = new ExceptionCatchingRequestBody(rawBody); try { T body = serviceMethod.toResponse(catchingBody); return Response.success(body, rawResponse); } catch (RuntimeException e) { // If the underlying source threw an exception, propagate that rather than indicating it was // a runtime exception. catchingBody.throwIfCaught(); throw e; } }

这个过程是在 serviceMethod 构造的时候，根据returnType判断，当前serviceMethod应该使用哪个convertAdapter进行 responseBody到 returnType的转换。

而ConvertAdapterFactory 也是在retrofit 构建的时候添加的。

最后，判断请求是否成功，然后作为参数传到ExecutorCallbackCall的enqueue方法中，然后将 步骤中的CallBack的onSuccess,onFailure方法通过callBackExecutor 传回主线程执行，到此，整个过程分析完毕

Retrofit

构造Retrofit：

baseUrl(String) 设置了基本url

convertAdapterFactories 其中将GsonConverter塞了进去

callBackExecutor 默认是一个Executor，里面有个成员变量是连接主线程looper的handler，当executor执行execute方法时候，将runnable对象通过handler发送到主线程执行，实现线程切换

adapterFactories 里面封装了一个默认的callAdapterFactory(callBackExecutor)，并且将callbackExecutor作为参数传了进去
