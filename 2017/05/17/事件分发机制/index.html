Android事件分发机制

参考：任玉刚《Android 开发艺术探索》

事件序列

当用户触碰了屏幕开始，就会产生一次事件序列

down事件 -> …( 0 到 n个Move事件) -> up事件

我们称用户从触碰屏幕到抬起手所触发的一系列事件成为一次事件序列。

当事件序列第一个 down事件进行分发的时候，首先到达的是 Activity，调用activity的dispatchMotionEvent( MotionEvent )

可以看到，当 PhoneWindow 对象消耗了这次事件，这次就会返回true，不会触发Activity的onTouchEvent事件

PhoneWindow它会调用顶层View - DecorView来处理分发事件。

而DecorView extends 于 FrameLayout，所以会出发 ViewGroup的 dispatchTouchEvent方法

在看源码的时候，我们需要理清一下事件分发的思路：

事件分发目的 遍历整个View树，找到可以处理 当前用户操作的事件序列的View控件，如果这个View接受处理，那么这一系列事件都直接交给它处理。

View树的结构是从上之下，一个View只有一个父View，而一个ViewGroup可以有多个View，并且ViewGroup本身也是一个View，本身也可以处理事件

基于View树的结构，在事件分发的时候，事件就会首先经过ViewGroup，如果ViewGroup不拦截的话，那么就将事件分发给它的子View处理。 如果ViewGroup进行事件拦截的话，那么应该调用ViewGroup本身的 onTouchEvent处理这次事件。如果ViewGroup设置了 onTouchListener 的话，则先调用 onTouchListener 的 onTouch事件，所以 onTouch 的优先程度是高于 onTouchEvent 的。

事件分发到View时，有两种结果：1. 返回了true，即消耗了事件，那么这一系列事件都交由它处理。如果返回false，则会调用父类ViewGroup的 onTouchEvent，向上递归处理。假如整个View树都不处理这次事件，那么就会回调Activity的 onTouchEvent。

但是假如View树中有View处理这次Down事件，那么我们就找到了这个View了，就将一系列事件交给它处理。但是考虑到优化，如果每次都要遍历View树，把这个View找出来，肯定是性能低下的，我们在父View中找到这个View时，view返回true，代表消耗这个事件，那么我们就在父view记下这个View，然后当前父View返回true，父view的父view又记下这个父view，那么一直往上记下来，Down事件后面的事件则不管其他情况，无论父View是否拦截的，都不用管，直接沿着这条索引链分发下来，到达目标View，交给它的onTouchEvent处理。

上面的注释已经非常详细地分析了整个ViewGroup分发的流程，我们可以得出下面几个结论：

  1. 事件序列只能被一个View拦截且消耗，处理了Down事件的View将会接收整个事件序列，并且后续所有事件是通过索引链直接沿 DecorView的顶层View下来，不用再次遍历View树，并且不会经过ViewGroup的拦截。
  2. 当某个ViewGroup一旦决定拦截Down事件，那么这一个事件序列都只能由它处理（如果事件序列能够传递给它的话），而不会继续调用它的 onInterceptTouchEvent 方法
  3. 如果一个View不处理Down事件，那么事件序列其他事件都不会经过它了，如果ViewGroup的所有子View都不处理Down事件，那么将交由ViewGroup自己处理，调用它的 onTouchEvent 方法，如果这个ViewGroup也不处理，一直递归向上，直到调用actviity的onTouchEvent方法。
  4. ViewGroup默认不拦截子View的onTouch事件分发，即 onInterceptTouchEvent 默认返回false，如果想要拦截，需要重写为true。

再来看看 onTouchEvent 的源码

从 onTouchEvent 的源码可以得出下面这几个结论：

  1. 如果绑定了 onTouchListener，那么将会优先调用listener的 onTouch方法，将事件交由listener处理。如果返回true，则事件消耗掉，不会继续分发到 onTouchEvent了。
  2. 假如没设置listener，当前View的状态尽管是 Disable，都不会影响 onTouchEvent的执行。并且只要View是支持clickable或者long_clickable的话，都会消耗这个事件序列，虽然不做什么事情。
  3. 如果View不是disable状态的话，那么在事件处理过程中会发生控件的onclick事件，所以 onTouchEvent是优先于onClick的。
