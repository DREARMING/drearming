okhttp源码分析

其实就是几个拦截器的事

Posted by MVCoder on April 28, 2017

Okhttp

参考博客：okhttp源码分析

在Android端，最出名的网络框架要数Retrofit了，而Retrofit本身不处理网络，而是由内部的网络请求框架 okhttp 进行处理，并且现在Android内部的网络框架 HttpUrlConnection 也是改为基于okhttp 实现，这样子，okhttp是当今最厉害的网络请求框架。

在Android客户端中，我们的网络框架可以抽象为哪些步骤呢？

  1. 构造一个请求Request（包括请求的Url，设置请求方法，参数，正文，请求头）
  2. 在client端发送请求到服务端（涉及到请求的缓存，取消，多线程分发，连接的重用管理）
  3. 从服务器端接收返回来的数据，封装成Response封装成 请求结果码，结果信息，响应头，响应正文）

看一下okhttp基本的请求：

Request

request是用 Builder 设计模式构建的，这种设计模式非常适用于参数非常多的的构造器。

可以看到，Request对象的的确高度封装成请求url、请求方法、请求头和请求正文这几个部分。非常简单，内部具体的细节，比如基本请求头的添加，method的默认是get等等。

okhttpClient

整个okhttp最核心的类，okhttpclient封装了所有核心的资源

注意到我们一开始发送请求时,调用这样的语句 client.newCall(Request)，OkhttpClient 实现了 Call.Factory 接口，实现了 newCall()方法，返回一个RealCall对象，而RealCall对象实现了Call接口，Call的源码：

可以看到，Call接口基本包含了我们请求时所需要的大部分操作，异步、同步、取消请求等。看一下Call的实现类RealCall如何实现这些操作；

我们看到，RealCall封装存储了 OkhttpClient和Request对象，当调用enqueue方法时，是通过okhttpclient的dispatcher,将封装了callback的AsynCall对象入队。

enqueue是异步请求的方法，而execute()则是同步请求的,这两个方法都是调用 okhttpclient的dispatcher来分发。来看看Dispatcher的实现：

我们可以看到，Dispatcher有三个队列，分别是运行中的异步队列，运行的同步队列，还有等待的异步队列，enqueue方法则是调用线程池来，执行AsynCall的execute方法（会判断当前请求访问的主机请求数大于阈值的话，则会要求等待）

execute则是丢进在当前的同步队列中，等待执行

回到 AsynCall 的execute方法看到，竟然直接返回Response对象了：

调用了getResponseWithInterceptorChain方法，下面是这个方法的代码：

我们可以看到，这里是一堆拦截器，并且拦截器是有顺序地添加进 interceptors 中，然后调用 RealInterceptorChain的procee方法返回请求结果 Response

可以看到，我们在 proceed 方法里面,我们根据构造时传递来的index，获取拦截器，调用其 intercept方法。这里又传进了一个 RealInterceptorChain 对象，不同的是，这个index+1了，所以我们可以知道，这里会递归地将所有 interceptor 执行一遍，然后返回Response对象。

这样我们的请求就经过一系列的拦截器，包括添加请求头，缓存管理，失败重连、重定向，发送请求到服务器这些操作，都由一系列拦截器动态的添加了，这种插拔式的方式，有效的对各个层面进行解耦，让程序扩展得更加容易。
