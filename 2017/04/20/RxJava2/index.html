RxJava 源码分析

操作符的和线程变换的原理

Posted by MVCoder on April 20, 2017

RxJava2

参考链接：Rxjava2源码分析

RxJava 是基于观察者模式搭建的第三方库，拥有数量庞大且功能极其强大的操作符集合，线程切换变得异常简单，Rxjava可以使程序的逻辑更加清晰。

基本例子:

Observable.create(new ObservableOnSubscribe<String>() { @Override public void subscribe(ObservableEmitter<String> e) throws Exception { e.onNext("1"); e.onComplete(); } }).subscribe(new Observer<String>() { @Override public void onSubscribe(Disposable d) { Log.d(TAG, "onSubscribe()"); } @Override public void onNext(String value) { Log.d(TAG, "onNext() called with: value = [" + value + "]"); } @Override public void onError(Throwable e) { Log.d(TAG, "onError() called with: e = [" + e + "]"); } @Override public void onComplete() { Log.d(TAG, "onComplete() called"); } }); log: onSubscribe() onNext() called with: value = [1] onComplete() called

首先我们需要清楚的是：

  1. 上游是如何发送数据的
  2. 下游是如何接收数据的
  3. 上游和下游是如何联系起来的
  4. 操作符的实现
  5. 线程的调度原理

create方法：

Observable.onCreate(new ObservableOnSubscribe<T>(){ @Override public void subscribe(ObservableEmitter<T> e) throws Exception { e.onNext("1"); e.onComplete(); } }); public static <T> Observable<T> create(ObservableOnSubscribe<T> source) { //..... return RxJavaPlugins.onAssembly(new ObservableCreate<T>(source)); }

以一个ObservableOnSubscribe接口作为参数，下面是它的接口声明

public interface ObservableOnSubscribe<T> { void subscribe(ObservableEmitter<T> e) throws Exception; }

ObservableEmiiter是一个发射器，它跟Observer有关，看一下它的声明：

//这是关于Disposable相关的操作 public interface ObservableEmitter<T> extends Emitter<T> { void setDisposable(Disposable d); void setCancellable(Cancellable c); boolean isDisposed(); ObservableEmitter<T> serialize(); } //我们可以发现，Emitter的接口跟我们Observer中的方法非常相似，我们在后面可以看到，ObservableEmitter将会对Observer进行包装，来确定上游和下游之间的关联 public interface Emitter<T> { void onNext(T value); void onError(Throwable error); void onComplete(); }

再返回看一下，在create方法内部，是通过ObservableOnSubscribe作为参数，构造了一个ObservableCreate对象；看看 RxJavaPlugins.onAssembly内部的构造：

@SuppressWarnings({ "rawtypes", "unchecked" }) public static <T> Observable<T> onAssembly(Observable<T> source) { Function<Observable, Observable> f = onObservableAssembly; if (f != null) { return apply(f, source); } return source; }

可以发现 ObservableCreate 本身是一个Observable，它是一个包装类，我们这里可以理解为直接返回它，即ObservableCreate对象

public final class ObservableCreate<T> extends Observable<T> { final ObservableOnSubscribe<T> source; public ObservableCreate(ObservableOnSubscribe<T> source) { this.source = source; } @Override protected void subscribeActual(Observer<? super T> observer) { CreateEmitter<T> parent = new CreateEmitter<T>(observer); observer.onSubscribe(parent); try { source.subscribe(parent); } catch (Throwable ex) { Exceptions.throwIfFatal(ex); parent.onError(ex); } }

ObservableCreate类继承与Observable，但是多了个subsribeActual，这个方法非常重要，是发射数据和接收数据，并且上下游建立关联的所在。

在ObservableCreate类中，做了如下步骤：

  1. 将创建的 ObservableOnSubscribe 对象 source 保存起来，这个对象是发射数据的源头，我们可以看看基本例子中，就是在它的subscribe方法里面，调用 ObservableEmitter 的onNext,onComplete方法.

  2. subscribeActual方法，我们先跳跃一下，看一下Observable对象的subscribe方法：

    //Observable实现了ObservableSource接口 public interface ObservableSource<T> { void subscribe(Observer<? super T> observer); }
    public final void subscribe(Observer<? super T> observer) {
    ... try { //1 hook相关，略过 observer = RxJavaPlugins.onSubscribe(this, observer); ... //2 真正的订阅处 subscribeActual(observer); } catch (NullPointerException e) { // NOPMD throw e; } catch (Throwable e) { //3 错误处理， Exceptions.throwIfFatal(e); // can't call onError because no way to know if a Disposable has been set or not // can't call onSubscribe because the call might have set a Subscription already //4 hook错误相关，略过 RxJavaPlugins.onError(e); NullPointerException npe = new NullPointerException("Actually not, but can't throw other exceptions due to RS"); npe.initCause(e); throw npe; } }

我们可以发现，这个方法内部实际是调用了subscribeActual（observer）方法,Observer作为参数传进去，observer是我们下游数据的接收端。它的接口声明如下：

public interface Observer<T> { void onSubscribe(Disposable d); void onNext(T value); void onError(Throwable e); void onComplete(); }

继续回到subscribeActual方法，我们就可以理解了上游和下游是如何分别发送和接收数据的，并且建立关联。

首先创建CreateEmitter发射器，并将Observer传进去，这里我们可以发现，其实createEmitter是一个包装类的角色

static final class CreateEmitter<T> extends AtomicReference<Disposable> implements ObservableEmitter<T>, Disposable { final Observer<? super T> observer; CreateEmitter(Observer<? super T> observer) { this.observer = observer; } @Override public void onNext(T t) { ... //如果没有被dispose，会调用Observer的onNext()方法 if (!isDisposed()) { observer.onNext(t); } } @Override public void onError(Throwable t) { ... //1 如果没有被dispose，会调用Observer的onError()方法 if (!isDisposed()) { try { observer.onError(t); } finally { //2 一定会自动dispose() dispose(); } } else { //3 如果已经被dispose了，会抛出异常。所以onError、onComplete彼此互斥，只能被调用一次 RxJavaPlugins.onError(t); } } @Override public void onComplete() { //1 如果没有被dispose，会调用Observer的onComplete()方法 if (!isDisposed()) { try { observer.onComplete(); } finally { //2 一定会自动dispose() dispose(); } } } @Override public void dispose() { DisposableHelper.dispose(this); } @Override public boolean isDisposed() { return DisposableHelper.isDisposed(get()); } }

当CreateEmtter发射数据的时候，先判断dispose是否为真，如果为真代表着下游已经取消订阅了。

并且onComplete和onError方法只能被调用一次，因为他们被调用后都会设置dispose为true；

onNext() 方法可以看到，我们CreateEmitter发射数据，会调用onNext方法，都是转为调用下游订阅者 observer 的onNext方法，onCompleted、onError也是一样，这是一种代理模式的存在。

看看subscribeActual的下一条语句：

observer.onSubscribe(parent);

parent是CreateEmitter对象，它实现了Disposable接口，所以当调用subscribeActual方法时，订阅者 observer 就会被调用 onSubscribe方法，并且将 createEmitter 传给 订阅者，这样子，订阅者就拥有了 控制上游发送数据的能力了。

再接下来是非常重要的一条语句，开始发射数据：

source.subscribe(parent);

source 是我们在 create()方法时传送进来的 ObservableOnSubscribe 对象，这里调用了它的 subScribe方法，并且将CrateEmitter对象作为参数传进去，这样，Observable就开始发送数据了。

调用流程

根据上面的过程，我们总结一下 一次简单的调用过程

  1. 我们首先通过 Observable的静态方法 create，传进一个 ObservableOnSubscribe对象作为参数
  2. 在create方法内部,创建了 ObservableCreate 对象，并且将 ObservableOnSubscribe 作为构造参数，保存起来。在ObservableCreate 继承于Observable，它有一个 subscribeActual 的方法 ，所以最后是，Observable的create方法，返回了一个ObservableCreate的对象
  3. 当下游的订阅者进行订阅时，会调用 Observable对象的 subscribe 进行订阅，subscribe内部调用 subscribeActual 方法，将 observer对象作为参数传进去。
  4. 在 subscribeActual 方法内部，将订阅者作为参数，构造 CreateEmitter 对象，在CreateEmitter内部，封装了 onNext、onCompleted、onError等方法，当调用CreateEmitter进行数据发射的时候，相当于一个代理对象，转为调用 observer 对象的onNext、onCompleted、onError方法。
  5. subscribeActual内部，还有一条语句，observer.onSubscribe(Disposable)，将CreateEmitter作为参数，调用订阅者的onSubscribe方法，使observer在下游具有控制上游发送数据的能力
  6. 最后调用 ObservableOnSubscribe对象的 subscribe方法，将 CreateEmitter 对象传进去，开始发送数据，因为发送数据是转接到 observer 的方法调用，所以实现了上下游的数据发送。

操作符原理

map 操作符用得非常频繁，用于将上游的数据做一次变换，拿它举个例子：

final Observable<String> testCreateObservable = Observable.create(new ObservableOnSubscribe<String>() { @Override public void subscribe(ObservableEmitter<String> e) throws Exception { e.onNext("1"); e.onComplete() } }); Observable<Integer> map = testCreateObservable.map(new Function<String, Integer>() { @Override public Integer apply(String s) throws Exception { return Integer.parseInt(s); } }); map.subscribe(new Observer<Integer>() { @Override public void onSubscribe(Disposable d) { Log.d(TAG, "onSubscribe() called with: d = [" + d + "]"); } @Override public void onNext(Integer value) { Log.d(TAG, "onNext() called with: value = [" + value + "]"); } @Override public void onError(Throwable e) { Log.d(TAG, "onError() called with: e = [" + e + "]"); } @Override public void onComplete() { Log.d(TAG, "onComplete() called"); } });

看看map操作符的源码：

public final <R> Observable<R> map(Function<? super T, ? extends R> mapper) { //判空略过 ObjectHelper.requireNonNull(mapper, "mapper is null"); //RxJavaPlugins.onAssembly()是hook 上文提到过 return RxJavaPlugins.onAssembly(new ObservableMap<T, R>(this, mapper)); }

这里跟上面的 ObservableCreate有点类似，返回了一个ObservableMap类型 的 Observable 对象。

public final class ObservableMap<T, U> extends AbstractObservableWithUpstream<T, U> { //将function变换函数类保存起来 final Function<? super T, ? extends U> function; public ObservableMap(ObservableSource<T> source, Function<? super T, ? extends U> function) { //super()将上游的Observable保存起来 ，用于subscribeActual()中用。 super(source); this.function = function; } @Override public void subscribeActual(Observer<? super U> t) { source.subscribe(new MapObserver<T, U>(t, function)); }

看到这里就非常熟悉了，同样是保存了source对象，不过subscrieActual不再创建了CreateEmitter对象，这样也很容易理解，因为map操作符只是将上游的数据进行转变，不涉及数据的创建。

我们可以知道，当下游订阅的时候，就会调用上一个Observable的subscribeActual方法，到了map操作符时，它是调用了其上面一个observable的subscribe方法，并且将下游的observer进行封装，封装成 MapObserver<T,U>,传递给上一个Observable；这样当上游发射数据的时候，就会调用MapObserver的onNext、onCompleted、onError 方法！

我们就清楚了 Map操作符起到了承上启下的作用。看一下 MapObserver的源码：

static final class MapObserver<T, U> extends BasicFuseableObserver<T, U> { final Function<? super T, ? extends U> mapper; MapObserver(Observer<? super U> actual, Function<? super T, ? extends U> mapper) { //super()将actual保存起来 super(actual); //保存Function变量 this.mapper = mapper; } @Override public void onNext(T t) { //done在onError 和 onComplete以后才会是true，默认这里是false，所以跳过 if (done) { return; } //默认sourceMode是0，所以跳过 if (sourceMode != NONE) { actual.onNext(null); return; } //下游Observer接受的值 U v; //这一步执行变换,将上游传过来的T，利用Function转换成下游需要的U。 try { v = ObjectHelper.requireNonNull(mapper.apply(t), "The mapper function returned a null value."); } catch (Throwable ex) { fail(ex); return; } //变换后传递给下游Observer actual.onNext(v); }

我们可以发现，onNext()方法里面，将上游传递下来的数据t，转换为v，在下面这句语句里面：

ObjectHelper.requireNonNull(mapper.apply(t), "The mapper function returned a null value.");

调用mapper，即map操作符构造的时候传进来 Function，它的apply方法，将数据t转换为我们需要的数据v；然后它持有下游的observer，调用其onNext方法，这样数据就经过了转换，传递给了下游的订阅者

总结一下操作符的操作流程

  1. 当下游订阅的时候，调用 map 操作符生成的Observable，即 ObservableMap，它的subscribeActual 方法，在里面，ObservableMap,它持有上游的Observable，调用其subscribe方法，并且将 封装了Function转换操作的 MapObserver的 observer 传到上游去。
  2. 当上游收到map操作符的调用，它会执行subscribe方法，调用其内部的 subscribeActual 方法，这个时候，如果这里就是源头了，这个subscribeActual将会有 CreateEmitter，封装map操作符传上来的 mapObserver，然后调用其 onSubscribe方法，最后开始发射数据。
  3. 源头发射数据的时候，实际调用了 mapObserver的 onNext 方法，在mapObserver的onNext方法里面，上游的数据，经过 function里面的操作转换，然后 mapObserver里面持有下游的observer，将经过转换的数据发送给下游，这样map的承上启下作用就完成了。

在这里我们可以发现，即使有多少个操作符，这种承上启下的作用都不会变换。上面的执行流程是一致的。

线程切换原理

Rxjava 除了让程序逻辑清晰的链式调用，还有极其方便的线程切换能力。

Observable.just(1, 2, 3, 4) // IO 线程，由 subscribeOn() 指定 .subscribeOn(Schedulers.io()) .observeOn(Schedulers.newThread()) .map(mapOperator) // 新线程，由 observeOn() 指定 .observeOn(Schedulers.io()) .map(mapOperator2) // IO 线程，由 observeOn() 指定 .observeOn(AndroidSchedulers.mainThread) .subscribe(subscriber); // Android 主线程，由 observeOn() 指定

subscribeOn

subscribeOn 可以设置发生订阅所在的线程，因为我们下游一旦订阅，就会沿着逻辑链向上调用上一个Observable的subscribe方法，直到上游源头，所以，订阅所在的线程就是数据发送所在的线程。因为逻辑链一直向上调用的原因，所以最接近上游的线程切换，才会真正决定订阅所在的线程。

看一下 subscribeOn 具体的源码：

public final Observable<T> subscribeOn(Scheduler scheduler) { //判空略过 ObjectHelper.requireNonNull(scheduler, "scheduler is null"); //抛开Hook，重点还是ObservableSubscribeOn return RxJavaPlugins.onAssembly(new ObservableSubscribeOn<T>(this, scheduler)); }

…我们发现，又是一段熟悉的代码，返回了一个 ObservableSubscribeOn 对象，这又是一个Observable类型，又是包装类的手法了

public final class ObservableSubscribeOn<T> extends AbstractObservableWithUpstream<T, T> { //保存线程调度器 final Scheduler scheduler; public ObservableSubscribeOn(ObservableSource<T> source, Scheduler scheduler) { //map的源码中我们分析过，super()只是简单的保存ObservableSource super(source); this.scheduler = scheduler; } @Override public void subscribeActual(final Observer<? super T> s) { //1 创建一个包装Observer final SubscribeOnObserver<T> parent = new SubscribeOnObserver<T>(s); //2 手动调用 下游（终点）Observer.onSubscribe()方法,所以onSubscribe()方法执行在 订阅处所在的线程 s.onSubscribe(parent); //3 setDisposable()是为了将子线程的操作加入Disposable管理中 parent.setDisposable(scheduler.scheduleDirect(new Runnable() { @Override public void run() { //4 此时已经运行在相应的Scheduler 的线程中 source.subscribe(parent); } })); }

ObservableSubscribeOn 保存了scheduler对象，然后在其 subscribeActual方法里面，创建了一个 SubscribeOnObserver 的包装类，封装来自下游的observer对象。

static final class SubscribeOnObserver<T> extends AtomicReference<Disposable> implements Observer<T>, Disposable { //真正的下游（终点）观察者 final Observer<? super T> actual; //用于保存上游的Disposable，以便在自身dispose时，连同上游一起dispose final AtomicReference<Disposable> s; SubscribeOnObserver(Observer<? super T> actual) { this.actual = actual; this.s = new AtomicReference<Disposable>(); } @Override public void onSubscribe(Disposable s) { //onSubscribe()方法由上游调用，传入Disposable。在本类中赋值给this.s，加入管理。 DisposableHelper.setOnce(this.s, s); } //直接调用下游观察者的对应方法 @Override public void onNext(T t) { actual.onNext(t); } @Override public void onError(Throwable t) { actual.onError(t); } @Override public void onComplete() { actual.onComplete(); } //取消订阅时，连同上游Disposable一起取消 @Override public void dispose() { DisposableHelper.dispose(s); DisposableHelper.dispose(this); } @Override public boolean isDisposed() { return DisposableHelper.isDisposed(get()); } //这个方法在subscribeActual()中被手动调用，为了将Schedulers返回的Worker加入管理 void setDisposable(Disposable d) { DisposableHelper.setOnce(this, d); } }

SubscribeOnObserver 直接调用下游 observer的方法，里面还设置了 disposable，当上游开始发送数据的时候，会调用下游的onSubscribe方法，传入CreateEmitter这个发送对象，属于Disposable，下游可以用于控制上游的数据发送，这里通过一些列处理来保持这种特性，当subScribeOnObserver的下游发生diapose的时候，那么这里这个包装类里面也会发生dispose，并且终止线程，同时调用上游的dispose；

回到 subscribeActual 方法里面，调用了scheduler.scheduleDirect(new Runnable) 开启子线程来 调用上游的 subscribe方法，并且把包装类 subscribeOnObserver传给上游。

/** * Schedules the given task on this scheduler non-delayed execution. * ..... */ public Disposable scheduleDirect(Runnable run) { return scheduleDirect(run, 0L, TimeUnit.NANOSECONDS); } public Disposable scheduleDirect(Runnable run, long delay, TimeUnit unit) { //class Worker implements Disposable ，Worker本身是实现了Disposable //在 subscribeOnObserver的setParent方法里面，封装了，当下游调用dispose方法时//候，这个work线程将会关闭； final Worker w = createWorker(); //hook略过 final Runnable decoratedRun = RxJavaPlugins.onSchedule(run); //开始在Worker的线程执行任务， w.schedule(new Runnable() { @Override public void run() { try { //调用的是 run()不是 start()方法执行的线程的方法。 decoratedRun.run(); } finally { //执行完毕会 dispose() w.dispose(); } } }, delay, unit); //返回Worker对象 return w; }

这里就是开启子线程调用了 上游的subscribe方法，我们可以知道，这样上游如果不进行其他线程切换的话，那么直到发送数据的时候，即订阅的时候，都会在当前指定的线程工作，所以就证明了一开始提出的结论， subscribeOn 决定了订阅发生的线程。并且，只有最接近上游的线程切换，才真正决定订阅所发生的线程

observerOn

我们在Android 中，经常会请求网络或者处理耗时操作后，更新UI，而更新Ui必须在主线程上，所以我们要切回主线程更新UI，而Rxjava是如何切换工作的线程呢？ 跟subscribeOn 不一样的是，subscribeOn决定的是发送数据所在的线程，但是在流式处理链上，数据的处理可能涉及到不同线程上，比如更新UI，而oberverOn则是指定处理过程中的线程，并且可以多次指定，不同过程发生在不同线程。

observerOn 发生的线程

public final Observable<T> observeOn(Scheduler scheduler) { return observeOn(scheduler, false, bufferSize()); } public final Observable<T> observeOn(Scheduler scheduler, boolean delayError, int bufferSize) { .... return RxJavaPlugins.onAssembly(new ObservableObserveOn<T>(this, scheduler, delayError, bufferSize)); }

又是这个套路，直接看 ObservableObserveOn 的代码

public final class ObservableObserveOn<T> extends AbstractObservableWithUpstream<T, T> { //本例是 AndroidSchedulers.mainThread() final Scheduler scheduler; //默认false final boolean delayError; //默认128 final int bufferSize; public ObservableObserveOn(ObservableSource<T> source, Scheduler scheduler, boolean delayError, int bufferSize) { super(source); this.scheduler = scheduler; this.delayError = delayError; this.bufferSize = bufferSize; } @Override protected void subscribeActual(Observer<? super T> observer) { // false if (scheduler instanceof TrampolineScheduler) { source.subscribe(observer); } else { //1 创建出一个 主线程的Worker Scheduler.Worker w = scheduler.createWorker(); //2 订阅上游数据源， source.subscribe(new ObserveOnObserver<T>(observer, w, delayError, bufferSize)); } }

一般都不会是TrampolineScheduler,所以会通过传进来的scheduler来创建一个worker，这里是主线程的scheduler，所以会创建一个运行在主线程的worker；

这样当上游发送数据下来， 那么就会调用 ObserverOnObserver 这个 observer对象的 onNext、onCompleted方法了。

再来看看 ObserveOnObserver的源码

这里的onNext方法并不是立刻发送数据了，而是将数据存在队列里面，然后调用worker来进行轮循queue中的数据，将其发送到下游，因为worker是指定了发生的线程，所以这里就发生了线程的切换，下游收到的数据，则是发生在 worker执行的线程。

所以得出这些结论：

  1. 在onNext()里，先不切换线程，将数据加入队列queue。然后开始切换线程，在另一线程中，从queue里取出数据，push给下游Observer
  2. observeOn()影响的是其下游的代码，且多次调用仍然生效。
  3. 关于多次调用生效问题。对比subscribeOn()切换线程是在subscribeActual()里做的，只是主动切换了上游的订阅线程，从而影响其发射数据时所在的线程。而直到真正发射数据之前，任何改变线程的行为，都会生效（影响发射数据的线程）。所以subscribeOn()只生效一次。observeOn()是一个主动的行为，并且切换线程后会立刻发送数据，所以会生效多次.
