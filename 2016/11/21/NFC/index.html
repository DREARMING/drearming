NFC是短距离传输技术，当我们支持NFC技术的手机触碰到NFC tag时，就会触发一个意图，试图找到对应的Activity，正因为是超短距离传输，我们应该默认地直接找到应用，而非存在多个Activity都能处理此Tag触发的意图，因为Activity Chooser被触发，我们要进行选择的话，很容易让我们的手机离开NFC TAG，从而导致连接失败的情况

如何定义一个Activity的Intent-filter，可以瞄准某类型的NFC TAG？

当支持NFC的Android手机检测到tag时，Intent-filter检查Action的顺序如下： 首先是 NEFT_DISCOVER -> TECH_DISCOVER ->TAG_DISCOVER

NDEF tag

首先一个NDEF tag的 NDEF Message 它有很多NDEF records，其中正确的形式封装，第一条NDEF Record里面应该有四个域存在：

    1. 一个 TNF 3bit 可以参照文档中的表格Table1，是指定tag的Type Name Format（类型名字格式，就是说你这个类型是什么，是一个uri，还是一个minetype，还是空的 等等），如果是well-know形式，会查看2中定义的类型（RTN）
    1. 一个 variable length type 参照Table2 ，定义tag确切的type类型（例如uri、text）
    1. 定义tag的id，不常用
    1. 负载在此条记录的数据（NDEF Message中有很多纪录，不要以为第一条纪录的数据就是全部）

###对于非NDEF数据的Tag，即类似饭卡那些不是标准的NDEF数据，如何筛选到**

if your activity filters for the ACTION_TECH_DISCOVERED intent, you must create an XML resource file that specifies the technologies that your activity supports within a tech-list set. Your activity is considered a match if a tech-list set is a subset of the technologies that are supported by the tag, which you can obtain by calling getTechList().

当tag发送的Intent的action是 ACTION_TECH_DISCOVERED ,必须定义一个tech-list的xml文档，当xml文档中有一个tech-list是tag所支持的技术的子集，就会被认为筛选到了，从而打开activity；我们可以通过getTechLIst()来获取tag中所支持的技术，获取到技术后，就能够通过这种技术的静态get(Tag tag)方法，对tag进行读写

AAR – Android Application Records 指定唯一应用能接受此tag （4.0以后）

我们就是要用这个tag打开固定的应用，如果应用存在就打开，否则就跳到google中下载，不能打开非此应用的activity；如果此应用有activity匹配tag中的筛选条件，那么就打开应用的这个activity，否则打开应用的mainactivity；

AAR记录写入，最好不要是第一条NdefRecord，第一条是用来标识uri和mine type的，除非你的应用只筛选 AAR
